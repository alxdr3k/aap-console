## 1. 개요

이 문서는 AAP Console 프로젝트의 PRD, HLD, ADR 등 설계 문서를 AI와 협업하여 작성한 과정에서 도출된 노하우를 정리한 것이다.

**핵심 원칙**: AI는 유능한 초안 작성자이지만, 도메인 전문가인 사용자의 지속적인 교정과 방향 제시 없이는 정확한 결과물을 낼 수 없다. AI와의 협업은 "지시 → 생성 → 검증 → 교정"의 반복 사이클이다.

## 2. 프로젝트 시작: 초기 프롬프팅

### 2.1 역할 부여 (Role Assignment)

대화 첫 문장에서 AI의 역할과 기대 수준을 명시한다.

```
너는 아주 유능한 소프트웨어 프로젝트 매니저야.
```

역할을 먼저 설정하면 AI가 해당 관점에서 일관된 어조와 깊이로 응답한다. "유능한"과 같은 수식어는 출력 품질에 실제로 영향을 미친다.

### 2.2 구조화된 초기 입력

첫 프롬프트에서 가능한 한 많은 맥락을 구조적으로 전달한다.

**효과적인 패턴:**
```markdown
# 업무 목표
1. 목표 A: 구체적 설명
2. 목표 B: 구체적 설명
...

---

# 참고사항
- 기술적 제약 조건
- 용어 정의
- 관련 시스템 정보

-> 위 업무목표는 별도의 문서로 저장
```

**핵심 포인트:**
- 마크다운 헤더(`#`)로 섹션을 분리하여 의미 단위를 명확히 한다
- 번호 매기기로 항목 간 순서와 계층을 표현한다
- `---` 구분자로 성격이 다른 정보를 분리한다
- 파일 저장 지시를 내용 안에 인라인으로 포함한다 (예: `-> 별도 문서로 저장`)

### 2.3 플래닝 우선 접근

AI가 바로 실행에 들어가면 방향이 틀어질 수 있다. 먼저 계획을 세우게 한다.

```
planing부터 다시 시작하자
```

계획 단계에서 AI의 이해도를 확인하고, 잘못된 방향을 잡았으면 실행 전에 교정한다. 잘못된 초안을 반복 수정하는 것보다 처음부터 다시 시작하는 것이 더 효율적일 수 있다.

## 3. 반복적 정교화 패턴

### 3.1 점진적 도메인 지식 공개 (Progressive Domain Revelation)

모든 것을 한 번에 설명하지 않는다. AI의 출력물을 보고 부족한 부분에 대해 점진적으로 도메인 지식을 추가한다.

**실제 사례 — 아키텍처 관계 교정:**

AI가 컴포넌트 간 관계를 잘못 그렸을 때, 올바른 관계를 설명하는 방식:

```
사실 각 app id가 독립적인 '인프라'를 가지지는 않아.
app이 생성되면 그 app을 지원하기 위한 langfuse org/project가 생성되고,
litellm에서 그 app으로 제공해줄 모델이나 가드레일 config 등이
litellm config(yaml)에 추가되는 것.
즉, 추가된 앱을 litellm, langfuse, keycloak등이 지원하기 위해
각 서비스들에게 정보를 업데이트해주는 거라고 보면 돼.
```

핵심은 "이렇게 고쳐"가 아니라 "실제로는 이런 구조야"라고 설명하는 것이다. AI가 근본 원리를 이해하면 관련된 다른 부분도 자동으로 교정한다.

### 3.2 교정 → 전파 → 검증 사이클

수정 사항을 반영한 후에는 반드시 전체 문서 일관성 검증을 요청한다.

```
여기까지 수정한 내용이 요구사항 등, 전체 문서에 틀어진 맥락과
디테일 없이 일관적으로 잘 반영되었는지 확인해
```

이 패턴은 "교정 → 전파 → 검증"의 3단계를 하나의 프롬프트에서 요청하는 것이다:
1. **교정**: 구체적 수정 지시
2. **전파**: 관련된 모든 섹션에 일관되게 반영
3. **검증**: 전체 문서의 정합성 확인

### 3.3 다중 항목 일괄 교정

관련된 여러 수정 사항은 대시(`-`)로 구분하여 하나의 메시지에 모아 전달한다.

```
- 각 서비스 설정 업데이트를 aap-helm-charts를 고쳐서 하는거야.
- 아키텍처 앞단의 keycloak에 '(B) terraform이 client 설정 업데이트'이건 빼
- s3 인증 정보는 어차피 하나로 돌려쓸거라서 앱이 추가될 때 버킷 경로 정도만 추가될거야.
- 3장의 '핵심 흐름'과 4장의 '설정 업데이트 모델'에 중복되는 내용이 있어.
```

여러 번 나눠서 보내면 AI가 매번 전체 문서를 다시 읽고 수정하므로 비효율적이다. 한꺼번에 보내면 모든 수정을 한 번의 패스에서 처리할 수 있다.

## 4. 설계 의사결정 프로세스

### 4.1 위임된 리서치 → 토론 → 결정 패턴

확신이 없는 기술적 결정은 AI에게 리서치를 위임한 뒤, 결과를 토론하고, 최종 결정은 사용자가 내린다.

**단계별 흐름:**

```
[1단계 - 리서치 위임]
"db를 사용하고 싶지 않은데, db를 대체할 수 있는 방법을 연구해봐"

[2단계 - 결과 평가 및 질문]
"오케이 좋아. 근데 dynamodb는 db 아니야?"
"근데 니가 위에 하이브리드 접근법이라고 추천한 걸로 하면 다 커버 되는거 아냐?"

[3단계 - 제약 조건 공개]
"db를 사용할 수 없는 환경이라서 그래. 어떻게든 방법을 강구해"

[4단계 - 최종 결정]
"오케이 그렇게 가자"
```

### 4.2 소크라테스식 질문법

결정을 내리기 전에 AI의 의견을 먼저 물어보면, AI가 사용자가 놓친 관점을 제시하거나 선택지를 정리해줄 수 있다.

```
"너는 어떻게 생각해?"
"뭐가 더 나은 선택인 것 같아?"
"어떻게 생각하는지 말해봐"
```

AI가 자신의 추론을 먼저 펼치게 하면, 그 추론의 결함을 잡아내기가 더 쉽다.

### 4.3 현실 제약 조건의 적시 공개

기술적으로 최적인 선택이 아니라 조직적·현실적 제약에 의해 결정이 달라지는 경우가 많다. 이런 제약을 적절한 시점에 공개하면 AI가 해당 제약 하에서의 최선을 찾는다.

**실제 사례:**

```
# 기술적 이유가 아닌 조직적 이유로 DB 회피
"db 서버를 채택하면 보안팀에게 보안검토를 받아야 하는게...
로컬 db는... 아키텍처에 드러나질 않고, 그래서 보안 검토를 우회할 수 있어서"

# 팀 역학 관계에 의한 기능 경계 설정
"keycloak은 다른 팀원이 구현해줘야 하거든.
근데 keycloak에서 많은 기능을 지원하는걸 달가워하지 않아"
```

AI는 조직 정치, 팀 역학, 관료적 프로세스를 알 수 없으므로, 이런 맥락을 명시적으로 알려줘야 현실적인 설계안이 나온다.

### 4.4 번호 매칭 응답

AI가 여러 옵션이나 질문을 제시할 때, 같은 번호 체계로 응답하면 모호함을 제거한다.

```
AI: "1. A안 vs B안? 2. 구현 범위는? 3. 보안 수준은? ..."

User:
1. A로 가자
2. 제안대로
3. 커스텀으로 구현했어
4. 고쳐
```

## 5. 오류 없는 문서를 위한 품질 보증

### 5.1 반복적 정합성 검증

한 번의 검증으로는 부족하다. "이슈 제로"가 될 때까지 반복 검증을 요청한다.

```
"문서 정합성 검증해. 디테일한 부분들이나 api 등등"
"이슈가 없을때까지 정합성 검증을 반복해"
```

AI는 한 번의 패스에서 모든 불일치를 잡아내지 못한다. 2~3회 반복하면 매 회차 새로운 이슈가 발견되는 것이 일반적이다.

### 5.2 교차 문서 검증

프로젝트에 여러 문서(PRD, HLD, ADR 등)가 있으면 문서 간 일관성도 검증해야 한다.

```
"console prd와의 정합성 검증해"
"config server의 prd, hld 문서 읽고 console prd와의 정합성 검증해.
 console prd에 config server의 디테일이 너무 많이 들어가지 않도록 해"
```

각 문서의 책임 범위(scope)를 명확히 지시하면 중복과 불일치를 동시에 방지한다.

### 5.3 문서 비대화 방지

AI는 대화 맥락의 부산물을 문서에 남기거나, 불필요한 세부사항을 추가하는 경향이 있다. 주기적으로 정리를 요청한다.

```
"문서에 더 이상 필요없는 내용이 있는지, 중복 내용이 있는지 탐색하고 처리해.
 문서가 너무 비대해"

"우리 대화 맥락을 문서에 적는 경우가 있는데
 그런 것들도 적절한 수준에서 제거해"
```

**문서 비대화의 주요 원인:**
- 대화 중 논의된 대안들이 최종 문서에 잔존
- 동일 내용이 여러 섹션에 중복 기술
- 다른 시스템의 내부 구현 세부사항이 인터페이스 수준을 넘어 포함
- "참고로", "이전에 논의한 바와 같이" 등 대화체 표현이 문서에 유입

### 5.4 형식 일관성 관리

여러 문서를 작성할 때 형식적 일관성도 중요하다.

```
"문서 형식 보고 차용할만한 좋은 좀 리스트업해. 내용 말고 문서 형식 말하는거야."
```

다른 잘 작성된 문서의 형식을 참고하면 자체 문서의 구조적 품질이 향상된다.

## 6. 효과적인 교정 기법

### 6.1 원인 설명형 교정 (Why, Not What)

"이걸 저걸로 바꿔"보다 "실제로는 이런 구조야"가 더 효과적이다.

```
# 비효과적 (무엇만 지시)
"SealedSecret을 LiteLLM 아래에서 빼"

# 효과적 (왜인지 설명)
"ss는 우리 ai assistant platform 서비스들이 운영되는 k8s 클러스터에서
 사용하는 거야. 거기에 ss controller가 각 서비스들이 사용하는 시크릿들을
 관리하는거야. 엄밀히 말하면 litellm에서 사용하는 환경변수가 ss로
 저장되어야 하지만, 저런 상하관계는 아니라는거지."
```

원인을 설명하면 AI가 관련된 다른 부분도 함께 수정하고, 같은 실수를 반복하지 않는다.

### 6.2 확인 질문을 통한 이해도 검증

교정 후 AI가 정말 이해했는지 확인 질문을 던진다.

```
"질문. 방금 수정하기 전에 terraform orch가 하던 일이 뭐야?"
```

AI가 변경 전 상태를 정확히 설명할 수 있어야 변경의 근거도 이해한 것이다.

### 6.3 간결한 승인 신호

만족스러운 결과에는 짧게 승인한다.

```
"그래"
"그래 전부 동의해"
"오케이 좋아"
"오케이 그렇게 가자"
```

불필요한 칭찬이나 부연 설명 없이 간결하게 승인하면 대화 흐름이 빨라진다.

### 6.4 반복 규칙의 명시적 강제

AI가 특정 규칙을 지속적으로 위반하는 경우가 있다. 반복해서 교정하되, 점점 더 직접적으로 표현한다.

```
# 첫 교정
"커밋 메시지 전부 영어로 바꿔"

# 재위반 시
"커밋 메시지는 영어로 작성하라니까?"
```

AI가 가장 잘 잊어버리는 규칙 유형:
- 커밋 메시지 언어 (영어/한글)
- 문서 범위 제한 (다른 시스템의 세부사항 포함 금지)
- 메타데이터 일관성 (버전, 날짜, 상태 등)
- 다이어그램 형식 규칙

이런 규칙은 CLAUDE.md 같은 프로젝트 설정 파일에 명시하면 세션 간에도 유지된다.

## 7. 문서 진화 관리

### 7.1 문서 생성 순서

이 프로젝트에서 문서는 다음 순서로 생성되었다:

```
1. business-objectives.md  ← 업무 목표 (원시 요구사항)
2. prd.md                  ← 제품 요구사항 정의서
3. development-process.md  ← 개발 프로세스 가이드
4. HLD.md                  ← High-Level Design
5. adr-001 ~ adr-006       ← Architecture Decision Records
6. ui-spec.md              ← UI 스펙 (초안)
```

**권장 순서의 근거:**
- 업무 목표가 먼저 정리되어야 PRD의 범위가 명확해진다
- PRD가 안정화된 후에 HLD를 작성해야 아키텍처와 요구사항이 일치한다
- ADR은 HLD 작성 과정에서 발생한 의사결정을 기록하므로 HLD와 병행하거나 이후에 작성한다

### 7.2 PRD의 진화 궤적

PRD는 12회 이상의 주요 리비전을 거쳤으며, 각 단계에서 전혀 다른 성격의 변경이 이루어졌다:

| 단계 | 변경 유형 | 예시 |
|------|----------|------|
| v1-v4 | 아키텍처 관계 교정 | SealedSecret, Keycloak, S3 위치 |
| v5-v6 | 개념 모델 전환 | "독립 인프라" → "공유 서비스 설정 업데이트" |
| v7-v8 | 신규 컴포넌트 추가 | Config Server 모듈 도입 |
| v9-v10 | 보안 모델 재설계 | Console의 시크릿 비저장 정책 |
| v11-v12 | 접근제어 경계 이동 | Keycloak RBAC → Console DB 자체 관리 |
| v13+ | 교차 문서 정합성, 비대화 제거 | 중복 제거, 대화 부산물 정리 |

**시사점**: 초기에는 "무엇이 맞는지" 교정에 집중하고, 후기에는 "문서가 깔끔한지" 정리에 집중한다.

### 7.3 컨텍스트 이어가기

긴 대화에서 AI의 컨텍스트가 소진되면 세션을 이어가야 한다. 이때 구조화된 요약이 핵심이다:

- 현재 진행 중인 작업
- 수정된 파일 목록
- 아직 해결되지 않은 이슈
- 다음 단계

## 8. 안티패턴과 주의사항

### 8.1 AI가 자주 틀리는 것들

| 유형 | 예시 | 대응 |
|------|------|------|
| **컴포넌트 관계 오류** | 피어(peer) 서비스를 상하 관계로 그림 | 실제 관계를 설명하고 전체 다이어그램 재검토 요청 |
| **ASCII 다이어그램 깨짐** | 화살표가 테이블을 관통, 박스 border 누락 | 구체적으로 어디가 깨졌는지 지적. 레이아웃 변경 요청 |
| **범위 초과 (Scope Creep)** | 다른 시스템의 내부 구현까지 상세 기술 | "인터페이스 수준만 기술하라"고 범위를 명시 |
| **대화 맥락 유출** | "이전에 논의한 바와 같이..." 표현 | 주기적으로 대화체 표현 제거 요청 |
| **과도한 세부사항** | 아직 결정되지 않은 구현 세부를 가정하여 기술 | "확정된 내용만 기술하라"고 제한 |

### 8.2 피해야 할 프롬프팅 패턴

```
# 나쁜 예: 모호한 지시
"문서 좀 다듬어줘"

# 좋은 예: 구체적 검증 기준 제시
"문서에 미스매치되는 부분, 내용이 잘못된 부분, 개선이 필요한 부분은 없는지 리뷰해"
```

```
# 나쁜 예: 정답을 직접 써줌
"3.2장의 세 번째 문장을 'X는 Y를 통해 Z한다'로 바꿔"

# 좋은 예: 원인을 설명하여 AI가 스스로 수정하게 함
"config server는 읽기 전용이 아니야. Console이 webhook을 보내면
 config server가 kubeseal → git commit & push → kubectl apply를 하는 구조야"
```

### 8.3 AI의 한계 인식

- **조직적 맥락 부재**: AI는 팀 역학, 보안 심의 프로세스, 내부 정치를 알 수 없다
- **암묵적 지식 부재**: "당연히 그렇게 하는 거 아니야?"라는 가정은 통하지 않는다
- **일관성 유지의 어려움**: 긴 문서에서 앞뒤 일관성을 AI가 자동으로 유지하기 어렵다
- **스타일 규칙 망각**: 커밋 메시지 언어, 문서 형식 등 반복 규칙을 쉽게 잊어버린다
- **다이어그램 정확도 한계**: ASCII 아트, Mermaid 등 시각적 요소는 반복적 수정이 필요하다

## 9. 실전 프롬프팅 레퍼런스

### 9.1 상황별 프롬프트 템플릿

**프로젝트 시작:**
```
너는 [역할]이야. 이 빈 레포에서 [프로젝트명] 프로젝트를 시작할거야.
아래 내용을 참고해서 [문서 유형]를 작성해.

# 업무 목표
1. ...
2. ...

# 참고사항
- ...
```

**아키텍처 교정:**
```
[컴포넌트 A]와 [컴포넌트 B]는 [현재 그려진 관계]가 아니야.
실제로는 [올바른 관계 설명].
[맥락/배경 설명].
```

**리서치 위임:**
```
[주제]에 대해 연구해봐. 특히:
1. [구체적 질문 1]
2. [구체적 질문 2]
각 접근법의 장단점과 실제 사용 사례를 포함해.
```

**정합성 검증:**
```
문서 정합성 검증해. [검증 범위 지정].
이슈가 없을때까지 정합성 검증을 반복해.
```

**비대화 제거:**
```
문서에 더 이상 필요없는 내용이 있는지, 중복 내용이 있는지 탐색하고 처리해.
대화 맥락이 문서에 들어간 부분도 적절한 수준에서 제거해.
```

### 9.2 전체 워크플로우 요약

```
┌─────────────────────────────────────────────────────┐
│  1. 역할 부여 + 구조화된 요구사항 전달                    │
│     └─▶ AI가 초안 생성                                │
├─────────────────────────────────────────────────────┤
│  2. 반복적 교정 사이클                                  │
│     ┌──────────────────────────────────┐             │
│     │  도메인 지식으로 교정               │             │
│     │  → 전파 (전체 문서 일관성)          │             │
│     │  → 검증 ("정합성 검증해")           │             │
│     │  → 추가 교정 필요 시 반복           │             │
│     └──────────────────────────────────┘             │
├─────────────────────────────────────────────────────┤
│  3. 설계 의사결정                                      │
│     리서치 위임 → 옵션 평가 → 제약 조건 공개 → 결정       │
├─────────────────────────────────────────────────────┤
│  4. 품질 보증                                         │
│     교차 문서 검증 → 비대화 제거 → 형식 통일              │
├─────────────────────────────────────────────────────┤
│  5. 다음 문서 단계                                     │
│     PRD ──▶ HLD ──▶ ADR ──▶ UI Spec                 │
│     (각 단계마다 2~4 반복)                              │
└─────────────────────────────────────────────────────┘
```

### 9.3 추가 노하우: AI가 알려주는 팁

위 내용은 사용자의 실제 경험에서 도출된 것이며, 아래는 추가적으로 권장하는 기법이다.

**1. Few-shot 예시 제공**
원하는 출력 형식의 예시를 직접 보여주면 형식 일관성이 크게 향상된다. "이런 식으로 써줘"와 함께 실제 예시 1~2개를 첨부한다.

**2. 네거티브 프롬프팅**
원하지 않는 것을 명시한다: "구현 세부사항은 쓰지 마", "이 문서에 Config Server 내부 로직은 포함하지 마".

**3. 체크리스트 기반 검증**
검증 시 구체적인 체크리스트를 제공하면 누락 없는 검증이 가능하다:
```
아래 항목을 검증해:
- [ ] 모든 FR 번호가 순차적인가
- [ ] 용어가 PRD 용어집과 일치하는가
- [ ] API endpoint가 HLD 스키마와 매칭되는가
- [ ] 다이어그램이 본문 설명과 일치하는가
```

**4. CLAUDE.md 활용**
프로젝트 루트에 CLAUDE.md 파일을 두면 세션이 바뀌어도 AI가 프로젝트 컨벤션을 기억한다. 커밋 메시지 언어, 문서 형식, 용어 규칙 등 반복적으로 위반되는 규칙을 여기에 기록한다.

**5. 단계별 확인점 설정**
대규모 작업은 중간 확인점을 설정하여 방향이 틀어지기 전에 교정한다:
```
"여기까지만 먼저 보여줘. 확인하고 다음 단계 진행할게."
```
